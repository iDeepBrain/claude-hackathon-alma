# Crisis Detection

## Design Philosophy

Crisis detection in Alma is **deliberately deterministic** — it does not use an LLM.

In a mental health context, the system that decides whether a user might be in crisis must be:
1. **Fast** — not dependent on LLM inference latency (100-800ms). A person in crisis cannot wait.
2. **Predictable** — the same message always produces the same score.
3. **Auditable** — a human can review and update the keyword list without understanding ML.
4. **Failure-safe** — if the MCP server is unreachable, the default is `{"score": 0.0}`, not an exception. The agent continues with the conversation.

## Implementation

The `evaluate_crisis_risk_tool` in `claude-hackathon-mcp/tools/crisis_tools.py`:

```python
# Returns a score from 0.0 to 1.0
# level: "none" | "low" | "moderate" | "high" | "critical"

evaluate_crisis_risk_tool(user_id: str, message: str) → {"score": float, "level": str}
```

Score thresholds:
- `0.0 – 0.3` → none / low: normal conversation
- `0.3 – 0.6` → moderate: Alma increases warmth and check-ins
- `0.6 – 0.8` → high: Alma asks direct safety questions
- `0.8 – 1.0` → critical: Alma provides crisis resources, offers to connect with human support

## When It Runs

Crisis evaluation runs **after the LLM response is streamed** — in the `asyncio.create_task(post_response_work)` background task in `AlmaChain`. It does not block the response.

If a high-risk score is detected, the state transitions to `crisis` mode, which:
1. Switches model routing to Opus 4.7 for all subsequent turns
2. Updates the system prompt with crisis-specific instructions
3. Adds the crisis event to the `mentioned_events` memory layer

## What Alma Says in Crisis Mode

From `prompts/alma_es.md`:
- Asks direct questions: "¿Estás pensando en hacerte daño?"
- Stays present: "Estoy aquí contigo en este momento"
- Provides resources without being dismissive
- Does NOT: diagnose, minimize, or offer generic advice

## Escalation Path

If crisis score exceeds 0.8, Alma proactively offers:
- MINSA Línea de Vida: 113 (Peru)
- Crisis Text Line equivalent for the user's region
- Option to connect with a volunteer listener (human)

This escalation is hardcoded in the prompt — it is not generated by the model, ensuring consistency.

---

## Proactivity Gate

Crisis detection serves a second purpose beyond conversation: **it gates all proactive outbound messages**.

Before the APScheduler sends any scheduled check-in (breakfast, lunch, or dinner), it queries the last known crisis score for the user:

```python
last_score = redis.get(f"alma:proactive:crisis_score:{user_id}")
if last_score and float(last_score) > 0.6:
    # suppress proactive message
    return
```

**Why this matters:** A user who has recently expressed high-risk thoughts should not receive a cheerful "¿Ya desayunaste? ☀️" message. This could:
- Break a fragile emotional state
- Feel dismissive of what they expressed
- Interrupt intentional silence (which can be a coping mechanism)

**The threshold is 0.6 (not 0.8):** We use the "high" threshold rather than "critical" because the proactivity gate is precautionary — better to skip a meal check-in than to risk interrupting someone in distress.

**This is the `emotional-companion` agent's most important contribution** to the architecture. It surfaced the "silence as self-regulation" principle in Round 2 of the multi-agent debate.

**The pipeline:**
```
APScheduler fires (08:30 / 13:30 / 19:30)
    ↓
Check alma:proactive:slot:{user_id}:{date}:{slot} → already sent? skip
    ↓
Check alma:proactive:last:{user_id} → active in last 2h? skip
    ↓
Check alma:proactive:crisis_score:{user_id} → crisis_score > 0.6? skip
    ↓
httpx.post(api.telegram.org/bot{token}/sendMessage, chat_id=..., text=...)
    ↓
redis.set(alma:proactive:slot:{user_id}:{date}:{slot}, "1", ex=86400)
```
